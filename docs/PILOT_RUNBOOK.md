# 首轮内部试运行执行手册

本手册用于把 Calcaneal Case 从原型推进到真实医生参与的首轮内部试运行。目标不是马上收集大量病例，而是先验证：医生愿不愿意用、上传和脱敏流程是否顺手、病例讨论是否有价值、哪些字段会增加负担。

## 试运行原则

- 只面向已确认身份的骨科医生和内部讲师。
- 当前阶段只收跟骨骨折方向的建议、脱敏教学病例思路和产品反馈。
- 不在 GitHub issue、公开截图、群聊或文档中上传可识别患者资料。
- 上传过程包含脱敏步骤；进入教学讨论、管理员可见或公开展示前，必须再次进行脱敏复核。
- 医生本地私密随访信息只用于本人随访，不进入公开讨论、GitHub issue 或共享样例。
- 共同存储上线前，不把当前原型中的 JSON 导出文件当作病例投稿文件流转。

## 首批人员安排

### 项目发起人

- 确定首批 2-4 位讲师名单。
- 确认每位讲师的 GitHub 用户名。
- 每周确定一个维护主题。
- 把医生反馈整理为 issue、文档或小版本改动。

### 内部讲师

- 每人每周完成 1 条有效反馈。
- 可以只评论 issue，不要求写代码。
- 优先反馈临床字段、影像流程、分型、随访、讨论区和隐私风险。

### 管理员候选人

- 先不开放真实管理员权限。
- 先评估病例是否适合教学讨论，提醒是否存在脱敏风险。
- 记录哪些讨论适合沉淀为教学材料。

### 技术维护者

- 把反馈拆成可执行的小改动。
- 每周至少整理一次 issue 和版本更新说明。
- 对涉及隐私、权限、研究、多中心和云端存储的改动做额外标记。

## 第一周任务

时间建议：2026-06-28 至 2026-07-05。

### 第 1 天：拉人进入

- 发出内部试用邀请。
- 让每位讲师注册 GitHub，并把用户名发给项目发起人。
- 让大家打开公开体验地址。
- 明确说明：不要上传真实可识别患者资料。

### 第 2-3 天：每人试一个流程

每位讲师任选一个方向：

- 病例录入是否符合医生习惯。
- 影像上传、归类、矫正、测量是否顺手。
- Essex-Lopresti、Sanders、Zwipp 分型是否好填。
- 随访节点和指标是否够用。
- 病例讨论区是否像真实教学场景。

### 第 4-5 天：提交反馈

每人至少完成一项：

- 新建 1 个 issue。
- 评论 1 个已有 issue。
- 提交 1 个完全脱敏的教学病例思路。
- 指出 1 个隐私风险或流程风险。

### 第 6-7 天：整理下一版

项目发起人和技术维护者完成：

- 归纳本周最重要的 3-5 个问题。
- 决定哪些进入下一版。
- 标记哪些需要法律、伦理、医院流程或信息安全进一步确认。
- 发布一段本周维护记录。

## 医生试用流程

1. 打开公开体验地址。
2. 进入医生工作台。
3. 新建一个测试病例。
4. 填写基础病例信息、分型、合并损伤和随访节点。
5. 上传脱敏后的影像，或使用不含患者身份信息的测试图片。
6. 尝试影像归类、矫正、滤镜、测量和保存测量图。
7. 尝试把病例设为不同可见范围，理解每种范围的含义。
8. 到 GitHub issues 写下哪里不顺手、哪里容易出错、哪里缺字段。

## 病例材料安全清单

以下内容不得进入公开 issue、公开截图或共享样例：

- 患者姓名、电话、身份证号。
- 住院号、门诊号、检查号、PACS 条码、二维码。
- 含患者身份的屏幕信息条。
- 可识别人脸或屏幕反光人物。
- 精确到可识别个人的罕见组合信息。
- 未经授权的完整病历、检查报告或原始影像资料。

可以提交的是：

- 完全脱敏后的教学病例思路。
- 不含患者身份信息的截图。
- 临床字段建议。
- 影像流程建议。
- 分型、测量、随访和讨论区建议。

## 当前阶段不要做

- 不做真实多中心研究入组。
- 不收患者真实身份进入共同存储。
- 不把 AI 建议当作诊断或手术决策。
- 不开放患者端。
- 不上传未脱敏原始影像到公开位置。

## 本轮成功标准

一周后，如果满足以下条件，就说明试运行有效：

- 至少 3 位医生完成真实反馈。
- 至少产生 6 条 issue 或评论。
- 至少整理出 3 个下一版必须改的问题。
- 至少确认 1 条隐私或脱敏流程的改进点。
- 至少形成 1 个适合未来公开教学的脱敏病例方向。
