# 自己如何维护这个项目

这份文档给项目发起人使用。你不需要每天写代码，真正重要的是维持节奏、收集反馈、让讲师参与、把有效意见沉淀成项目更新。

最简单的维护方式就是把 GitHub 当成项目讨论板：你每周在仓库里发布一个 `issue`，讲师和维护者在下面评论。能变成产品改动的评论，再整理成下一版任务。

## 每周只做 5 件事

### 1. 发一个本周主题

每周选一个小主题，不要太大。

例子：

- 本周讨论病例录入字段
- 本周讨论影像上传和四点矫正
- 本周讨论 Böhler 角和 Gissane 角测量
- 本周讨论 Essex-Lopresti / Sanders / Zwipp 分型
- 本周讨论随访指标和终极指标

### 2. 让讲师每人做一件小事

每位讲师每周只需要：

- 评论 1 个 issue
- 提 1 条临床建议
- 提 1 个脱敏教学病例思路
- 指出 1 个隐私风险
- 指出 1 个界面不好用的地方

不要让他们感觉这是额外工作。每人 5-15 分钟即可。

### 3. 你做一次判断

每周看一遍讲师反馈，分成 4 类：

- 立即要改
- 后续再做
- 需要医学讨论
- 涉及隐私/伦理/法律，先暂停

### 4. 让 Codex 帮你整理

你可以直接对 Codex 说：

```text
把这周几个讲师的反馈整理成 GitHub issue 和下一版任务清单。
```

或者：

```text
根据这些反馈，帮我更新 README / ROADMAP / 原型页面。
```

### 5. 每 1-2 周做一次小发布

小发布不需要很大，可以只是：

- 新增一个字段
- 调整一个说明
- 修一个按钮
- 补一个文档
- 整理一批 issue

关键是让项目一直有维护痕迹。

## 最小维护节奏

如果你很忙，只做这三件事也可以：

1. 每周发一个主题
2. 每周让讲师评论一次
3. 每周让 Codex 整理一次

## 申请 Codex for Open Source 前的最低标准

建议至少有：

- 公开 GitHub 仓库
- 2-4 位讲师参与
- 10 个以上 issues
- 10 条以上讲师或维护者评论
- 3-5 次真实提交
- README、Roadmap、Governance、Security、Contributing 完整
- 一个 `v0.1.0` release

如果 10-14 天内能做到这些，就可以尝试申请。

## 你真正要负责的事

你不需要把每个细节都自己做完。

你真正要负责：

- 方向
- 医学质量
- 三步复位法的核心表达
- 找到愿意参与的讲师
- 保持项目节奏
- 判断哪些内容可以公开、哪些必须保护隐私

其他很多执行工作，都可以让讲师、维护者和 Codex 一起帮你完成。
