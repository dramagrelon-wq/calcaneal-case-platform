# 共同存储第一阶段方案

当前网页版本以本机浏览器存储为主，适合原型体验、流程讨论和开源维护。真正开始收集病例时，需要增加后端、权限、文件存储、审核和日志。第一阶段建议先做“手机可用的去标识化教学病例库”，不要一开始就收集患者真实身份信息进入云端。

## 第一阶段目标

- 手机端优先可用。
- 医生实名注册和资质审核。
- 上传即进入脱敏流程。
- 教学讨论、管理员可见和公开展示前进行二次复核。
- 共同存储只保存去标识化后的病例材料。
- 患者姓名、电话、身份证号等私密随访信息仍留在医生本人私密区，暂不进入共同云端。

## 推荐产品形态

先做同一套后端，前端分阶段推进：

1. 手机网页/PWA：最快落地，适合当前试运行。
2. 微信小程序：适合国内医生传播和日常入口。
3. 独立 App：适合后续更强的相机、离线、加密存储和推送能力。

这三个入口不要做成三套数据系统。病例、影像、讨论、权限和审核都应进入同一个后端。

## 后端模块

### 账号与准入

- 实名注册。
- 执业医师证、职称证和单位信息上传。
- 管理员人工审核。
- 显示名和真实身份分离。
- 账号封禁、权限回收和审核记录。

### 病例数据库

- 病例基础字段。
- 分型记录。
- 手术方式。
- 合并伤和伴发疾病。
- 随访节点。
- 讨论记录。
- 未读讨论数。

### 影像文件存储

- 原图进入临时处理区。
- 脱敏和压缩后进入正式病例影像区。
- 分类标签支持一张图多个用途。
- 支持术前、术中、术后和随访时间点。
- 测量后的标记图单独保存，关联原图。

### 脱敏审核

- 上传时提示医生完成清理。
- 系统自动检查常见敏感信息。
- 公开、圈内教学、管理员可见前进行二次复核。
- 审核不通过时，要求重新清理或改为私密。
- 管理员可以反馈漏检类型，后续优化规则。

### 权限模型

- 私有：仅上传医生可见。
- 管理员可见：上传医生和认证管理员可见。
- 圈内教学：认证医生范围内讨论。
- 公开教学：脱敏后可展示给更广泛医生。
- 多中心研究：单独资格、伦理和项目流程，暂不默认开放。

### 日志与追责

- 登录记录。
- 上传、修改、删除、恢复记录。
- 可见范围变更记录。
- 讨论和管理员处理记录。
- 违规内容处理记录。

## 第一版不要云端保存的内容

- 患者真实姓名。
- 电话号码。
- 身份证号。
- 住院号、门诊号、检查号。
- 未脱敏原始影像。
- 未经伦理审批的多中心研究原始数据。

## 需要你准备的资料

### 人员资料

- 首批试用医生名单。
- 管理员候选人名单。
- 每个人的 GitHub 用户名。
- 后续真实系统中的医生身份审核标准。

### 内容资料

- 10-20 个完全脱敏的教学病例方向。
- 三步复位法的标准表达。
- 常见跟骨骨折分型图示或授权材料。
- 手术方式和随访指标的标准词表。

### 合规资料

- 医生准入告知。
- 病例上传与脱敏承诺。
- 隐私政策。
- 平台使用规则和违规追责条款。
- 多中心研究流程占位说明。

### 技术资料

- 是否使用国内云服务。
- 是否需要独立域名。
- 是否需要医院或团队主体。
- 未来是否要接短信、微信登录、微信订阅消息或小程序。

## 第一阶段验收标准

- 手机上可以完成注册、审核、建病例、上传影像、归类、脱敏确认和讨论。
- 医生不能看到自己无权访问的病例。
- 公开或教学病例不含可识别患者信息。
- 管理员可以处理风险病例和讨论。
- 每次可见范围变更都有记录。
- 删除病例进入垃圾箱，并可以在保留期内恢复。
